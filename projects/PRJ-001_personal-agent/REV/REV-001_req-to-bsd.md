## 1. レビューサマリー

REQ フェーズ（REQ-001〜REQ-008）と BSD フェーズ（BSD-001〜BSD-007、BSD-009、BSD-010、BSD-008）間の整合性レビューを実施した。

**全体評価**: BSD 成果物は全体的に高品質であり、REQ の主要要件を適切に反映している。Critical 問題は検出されなかった。

**主な発見事項**:
1. **【重要】BSB-003（画面設計書）にログイン画面（SCR-001）が設計されているが、REQ フェーズでは「シングルユーザー・認証不要」と定義されており（BSD-001・BSD-002）、設計方針との矛盾が存在する。** これは Major 問題として記録する。
2. BSD-005（API基本設計書）でも認証エンドポイント（`/api/v1/auth/login` 等）が定義されており、REQ-004 の非機能要件（シングルユーザー・ローカル環境のため認証不要：NFR-SEC-05、CON-B-01 参照）との不整合がある。
3. BSD-004（業務フロー設計書）に「BF-005: ログイン認証」フローが設計されており、REQ-002 のビジネス要件（認証なし）との乖離がある。
4. BSD-007 に `delete_issue` ツールが定義されているが、REQ-002 の「BR-02: タスク削除操作はエージェントが行わない」というビジネスルールとの整合性が要確認である（Minor）。
5. FEAT-001〜FEAT-006 の全 FEAT-ID が BSD の各成果物に適切にカバーされており、機能の網羅性は良好。

**注記**: このプロジェクトの REQ ドキュメント番号体系は review-criteria.md が想定するものと異なる（例：REQ-005 がユースケース一覧・FEAT-ID 一覧を兼ねる、REQ-004 が非機能要件定義書）。以下のレビューではプロジェクト実際のドキュメントに基づいて評価を実施した。

## 6. 推奨事項

1. **【最優先】認証要否の設計方針統一（ISS-001）**: DSD フェーズ開始前に、フェーズ1での認証実装の要否について明確な決定を行うこと。REQ-004（NFR-SEC-05: ローカル通信のみ・認証不要）と BSD-003（SCR-001ログイン画面設計）の方針を統一し、BSD-003・BSD-004・BSD-005 を整合させること。フェーズ1では認証なしとするのであれば、該当箇所を OI（未確認事項）または「フェーズ2スコープ」として明記することを推奨する。

2. **【高優先】ビジネスルールの下流設計への明記（ISS-002）**: BSD-004 の各業務フロー定義に REQ-002 の BR-01・BR-02 を「ドメイン不変条件」として追記すること。特に「タスク削除はエージェント経由では禁止（BR-02）」は DSD フェーズ・IMP フェーズでの実装時に重要な制約となるため、設計書に明記しておくことが必須である。

3. **【中優先】削除ツールの利用制約注記（ISS-003）**: BSD-007 の `delete_issue` MCP ツール定義に BR-02 に基づく利用制約注記を追加すること。DSD-005 での MCP アダプター実装時に誤った実装を防ぐための設計上のガードレールとして機能させること。

4. **（参考）ドキュメント命名体系の整備**: このプロジェクトの REQ ドキュメント番号体系（REQ-003 = 機能要件定義書、REQ-004 = 非機能要件定義書等）は review-criteria.md の標準体系と異なる。次フェーズ以降のレビュー作業の効率化のために、document-list.md でプロジェクト固有のドキュメント体系を明文化しておくことを推奨する。
