## 1. セキュリティ設計方針

### 1.1 目的・基本方針

本システムはローカルPC上で動作する個人利用AIエージェントシステムである。外部公開・マルチユーザー対応は行わない（CON-B-01、CON-B-03）。セキュリティ設計の主眼は以下の2点に置く。

1. **APIキー・認証情報の漏洩防止**: Anthropic APIキー・Redmine APIキーを安全に管理し、バージョン管理システム（Git）への混入を防ぐ。
2. **意図しないデータ露出防止**: 個人情報・機密情報のログ出力防止と、タスク誤削除防止。

ローカル環境・個人利用の特性上、通信暗号化（外部APIを除く）・認証認可の要件は最小限とする。

### 1.2 準拠標準・規制

- **準拠するセキュリティ標準**: OWASP Top 10 のうち、ローカルアプリケーションに適用可能な項目（特に機密情報管理・インジェクション対策）を参考にする。
- **法的要件**: 個人利用・日本国内のみ。個人情報保護法の対象外（個人利用のため）。ただし、Anthropic Claude API にはユーザーが入力したタスク内容等が送信される（BR-03）ことをユーザーが認識した上で利用する。

## 2. 認証・認可設計

### 2.1 認証方式

| 方式 | 採用 | 詳細 |
|---|---|---|
| セッション認証 | × | シングルユーザー・ローカル環境のため不要（CON-B-01） |
| JWT（Bearer Token） | × | 同上 |
| OAuth 2.0 / OIDC | × | 同上 |
| APIキー（Redmine向け） | ○ | X-Redmine-API-Key ヘッダーでRedmineへの認証を行う。環境変数 REDMINE_API_KEY で管理。 |
| APIキー（Claude API向け） | ○ | x-api-key ヘッダーでAnthropic APIへの認証を行う。環境変数 ANTHROPIC_API_KEY で管理。 |

**フロントエンド↔バックエンド間**: ローカルホスト内通信のため認証なし（NFR-SEC-05）。フェーズ2でクラウド化が必要になった場合は再設計する。

### 2.2 ロール・権限設計

個人利用・シングルユーザーのため、ロール設計は不要（CON-B-01）。

| ロール | 概要 | アクセス可能リソース |
|---|---|---|
| ユーザー（唯一） | システムの唯一の利用者 | 全機能（タスクCRUD・チャット・ダッシュボード）。ただしタスク削除はエージェント経由では不可（BR-02）。 |

### 2.3 セッション管理

- **セッション有効期限**: ブラウザセッション内のみ（OI-03: 会話履歴の永続化は基本設計フェーズで確認中）。フェーズ1ではインメモリで管理。
- **リフレッシュトークン方針**: 認証機能なし。不適用。
- **多重ログイン制御**: シングルユーザーのため不適用。
- **ログアウト処理**: 認証機能なし。不適用。

## 3. 通信暗号化

| 通信経路 | プロトコル | TLSバージョン | 証明書管理 |
|---|---|---|---|
| ブラウザ ↔ フロントエンド（localhost:3000） | HTTP | 不適用（ローカルのみ） | 不要（NFR-SEC-05） |
| フロントエンド ↔ バックエンド（localhost:8000） | HTTP | 不適用（ローカルのみ） | 不要（NFR-SEC-05） |
| バックエンド ↔ Redmine（localhost:8080） | HTTP | 不適用（ローカルのみ） | 不要（CON-T-07） |
| バックエンド ↔ Anthropic Claude API（外部） | HTTPS | TLS 1.2以上（Anthropic側で管理） | Anthropic側で管理。Pythonの requests / httpx ライブラリが標準で検証。 |

**重要**: 外部通信はAnthropicへのHTTPSのみ。その他はすべてローカルホスト内通信。

## 4. データ保護方針

### 4.1 保管データの暗号化

- **APIキー（Anthropic・Redmine）**: `.env` ファイルで管理。平文保存だが `.gitignore` 登録（CON-S-02）により外部漏洩を防ぐ（NFR-SEC-01）。OSのファイルシステムアクセス制御に依存。
- **個人情報**: 本システムが直接管理する個人情報データベースはなし（フェーズ1）。Redmineに保存されるタスク内容はRedmineのDBに保管。
- **会話履歴**: フェーズ1ではインメモリ管理のため保管なし。DB保存する場合（フェーズ2以降）は暗号化を検討する（OI-03）。
- **機密データ**: タスク内容にパスワード等が含まれないよう、ユーザーへの注意喚起を行う（BR-03）。

### 4.2 個人情報管理

- **取得する個人情報の種類**: ユーザーが入力したタスク内容・チャット内容（Anthropicに送信される）。
- **保存期間・廃棄方針**: Redmineに保存されたチケット情報はユーザーがRedmine上で管理。チャット履歴はフェーズ1ではセッション終了で消去。
- **アクセスログ管理**: エージェントの処理ログをファイルに記録（NFR-MAINT-01）。ログには個人情報・機密情報を平文出力しない（NFR-SEC-04）。

## 5. 脆弱性対策方針

| 脅威 | 対策 |
|---|---|
| SQLインジェクション | Redmineへのアクセスは公式REST APIを経由するためSQLを直接実行しない。バックエンドにDBを持つ場合はORMを使用する。 |
| XSS | Next.jsのReactベースレンダリング（自動エスケープ）を使用。エージェント返答をMarkdownレンダラー（react-markdown等）で表示する際はHTMLサニタイゼーションを適用する。 |
| CSRF | ローカルホスト環境のため外部からのCORSリクエストは発生しない。フロントエンドからのリクエストのみを受け付けるCORS設定をFastAPIに適用する。 |
| 不正アクセス | ローカルホスト内のみで動作。外部からのアクセスは物理的に不可（CON-T-07）。 |
| 機密情報漏洩（APIキー） | 環境変数管理（NFR-SEC-01）・.gitignore登録（NFR-SEC-02・CON-S-02）。定期的なキーローテーションを推奨（RISK-O-02対策）。 |
| 機密情報漏洩（ログ） | ログにAPIキー・タスク内容（個人情報含む可能性）を平文出力しない（NFR-SEC-04）。ログレベルに応じたマスキングを実装する。 |
| 依存ライブラリ脆弱性 | `requirements.txt` および `package.json` でバージョンをピン留め（NFR-MAINT-03、NFR-MAINT-04）。定期的な `pip audit` / `npm audit` の実施を推奨する（RISK-T-02対策）。 |
| タスク誤削除 | エージェントを通じたタスク削除操作を禁止（BR-02）。削除はRedmine Web UIからの手動操作のみとする。 |
| Prompt Injection | ユーザー入力をそのままプロンプトに埋め込む際のインジェクションリスクを認識する。システムプロンプトで操作範囲を明示的に制限する設計とする。 |

## 6. セキュリティヘッダー設定

ローカル環境・個人利用のため、プロダクションレベルのセキュリティヘッダーは**フェーズ1では省略**。ただし、FastAPIのCORS設定のみ適用する。

```python
# FastAPI CORS設定（バックエンド）
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:3000"],  # フロントエンドのオリジンのみ
    allow_methods=["GET", "POST"],
    allow_headers=["Content-Type"],
)
```

フェーズ2でクラウド公開が必要な場合は以下のヘッダーを追加する（要確認）:

```http
Content-Security-Policy: default-src 'self'; script-src 'self'
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Strict-Transport-Security: max-age=31536000; includeSubDomains
```

## 7. セキュリティ監査・ログ

- **認証ログ（成功・失敗）**: 認証機能なし。不適用（フェーズ1）。
- **操作ログ**: エージェントのツール呼び出しログを記録（FR-AGT-05・NFR-MAINT-01）。対象操作: タスク作成・更新・検索のMCP呼び出し全般。保存先: `logs/agent.log`（ローカルファイル）。保存期間: ローカル環境のためユーザーが任意に管理。
- **異常検知**: ローカル環境のためリアルタイム監視は不要。ログファイルの目視確認でデバッグ対応する。
- **ログフォーマット**: INFO / DEBUG / ERROR の3レベル対応（NFR-MAINT-01）。ログにはAPIキーを含む環境変数の値を出力しない。タスク内容を出力する場合は先頭50文字程度に切り詰める。
